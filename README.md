# TBN-IMCP4

IMC Prosperity 4 algorithmic trading competition entry.

## Prerequisites

- Python 3.11+
- Node.js 18+ — [nodejs.org](https://nodejs.org)
- pnpm — `npm install -g pnpm`
- [`prosperity4btx`](https://pypi.org/project/prosperity4btx/) backtester — `pip install prosperity4btx`

## First-time setup

```bash
git clone --recurse-submodules <repo-url>
cd TBN-IMCP4
python setup_visualizer.py   # installs and builds the local visualizer (once)
```

If you already cloned without `--recurse-submodules`:
```bash
git submodule update --init --recursive
python setup_visualizer.py
```

## Running a Backtest

```bash
python run_backtest.py 0                               # default: viz/message_viz.py
python run_backtest.py 0 --algo viz/attempt2_viz.py   # specific strategy
python run_backtest.py 0 --merge-pnl                   # merge P&L across days
```

This will:
1. Run the backtest → saves `output.log`
2. Start a local server on port `8765` serving both the visualizer and log
3. Open the visualizer in your browser with results pre-loaded

Press `Ctrl+C` to stop the server.

### Manual (no visualizer)

```bash
prosperity4btx viz/message_viz.py 0 --out output.log
```

## IMC Submission vs. Logger Boilerplate

### What the IMC platform template gives you

```python
logger = Logger()

class Trader:
    def run(self, state: TradingState) -> tuple[dict[Symbol, list[Order]], int, str]:
        result = {}
        conversions = 0
        trader_data = ""   # plain empty string

        # TODO: Add logic

        logger.flush(state, result, conversions, trader_data)
        return result, conversions, trader_data
```

### Key differences in this repo's implementation

| | IMC template | This repo |
|---|---|---|
| `trader_data` type | `str` (empty, unused) | `dict` loaded from JSON, serialized back at end |
| `run()` body | stub | full strategy logic |
| `logger.flush()` argument | `trader_data = ""` | `traderData = json.dumps(trader_data)` |

### Logger rules

- **Do not modify** the `Logger` class — required verbatim for the visualizer to parse output
- **Always call** `logger.flush(...)` as the last statement before `return` in `_viz.py` files
- **Use `logger.print()`** instead of `print()` for debug output in `_viz.py` files
- The `Logger` class must be present in the submitted file — IMC does not provide it at runtime
- `_submit.py` files have **no Logger** — they're clean for direct upload

### Adding a new strategy

1. Create `viz/{name}_viz.py` — full Logger boilerplate + strategy logic
2. Create `submit/{name}_submit.py` — same logic, Logger removed, `logger.print()` → `print()`
3. Test: `python run_backtest.py 0 --algo viz/{name}_viz.py`
4. Submit: upload `submit/{name}_submit.py` to the IMC platform

## Visualizer

The visualizer (`visualizer/`) is a git submodule pointing to [jmerle/imc-prosperity-3-visualizer](https://github.com/jmerle/imc-prosperity-3-visualizer). Built locally by `setup_visualizer.py`, served entirely from Python — no internet required at runtime.

`run_backtest.py` routes a single server on port `8765`:
- `/imc-prosperity-3-visualizer/` → `visualizer/dist/`
- `/output.log` → project root `output.log`

Use `logger.print(...)` in `_viz.py` files to emit debug messages visible in the **Lambda Logs** panel.
